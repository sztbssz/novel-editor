# 小说编辑器与飞书菜单集成方案分析

## 三种方案对比

### 方案A：桌面版监听HTTP请求（推荐）⭐

**实现原理：**
桌面版启动一个本地HTTP服务器（端口可配置，如18790），OpenClaw收到飞书菜单事件后，发送HTTP POST请求到桌面版，桌面版收到请求后弹出/聚焦窗口。

**优点：**
- ✅ 实现简单，只需一个HTTP endpoint
- ✅ 响应快速，无中间环节
- ✅ 桌面端自主控制窗口行为
- ✅ 可扩展性强，未来可支持更多远程控制功能
- ✅ 不依赖外部网络，纯本地通信

**缺点：**
- ⚠️ 需要桌面版保持运行（但用户本来就是点击菜单启动，隐含此前提）
- ⚠️ 端口占用问题（可配置解决）

**复杂度：** ⭐⭐（低）

---

### 方案B：WebSocket实时通信

**实现原理：**
桌面版启动时连接OpenClaw Gateway的WebSocket，建立长连接。菜单事件通过Gateway转发到桌面版。

**优点：**
- ✅ 双向实时通信
- ✅ 可推送更多状态信息
- ✅ 连接保持后可复用

**缺点：**
- ❌ 实现复杂，需要WebSocket服务端和客户端
- ❌ 需要处理连接断开、重连逻辑
- ❌ 需要修改Gateway核心代码
- ❌ 过度设计，当前需求只需单向触发

**复杂度：** ⭐⭐⭐⭐（高）

---

### 方案C：卡片消息 + URL Scheme

**实现原理：**
点击菜单，机器人回复卡片消息，用户点击卡片按钮，通过自定义URL scheme（如`novel-editor://open`）启动本地应用。

**优点：**
- ✅ 无需桌面版常驻
- ✅ 利用系统原生机制

**缺点：**
- ❌ 需要额外点击一次（菜单→卡片→按钮）
- ❌ 需要注册URL scheme（系统级配置）
- ❌ Windows/macOS/Linux配置方式不同
- ❌ 浏览器安全限制可能阻止自定义scheme

**复杂度：** ⭐⭐⭐（中）

---

## 推荐方案：方案A（HTTP监听）

**核心理由：**
1. **简单可靠** - HTTP是最稳定的本地通信方式
2. **快速响应** - 用户点击菜单→OpenClaw发送请求→桌面版弹出，一气呵成
3. **可维护** - 代码量少，易于调试和扩展
4. **无依赖** - 不依赖浏览器、系统scheme等外部因素

## 通信协议

### 请求格式
```http
POST http://127.0.0.1:18790/api/open
Content-Type: application/json

{
  "action": "open_creator",
  "timestamp": 1712651234567
}
```

### 响应格式
```json
{
  "success": true,
  "message": "Window opened"
}
```
